Node.js 한국 사용자 그룹

아래는 NodeJS로 작성된 Web Server의 예로, 모든 요청에 대해 "Hello World"로 응답을 합니다. 

var http = require('http');
http.createServer(function (req, res) {
  res.writeHead(200, {'Content-Type': 'text/plain'});
  res.end('Hello World\n');
}).listen(1337, "127.0.0.1");
console.log('Server running at http://127.0.0.1:1337/');

서버를 실행하기 위해서는 위의 코드를 example.js 파일로 작성한 후, 아래와 같이 node 명령을 실행하세요 : 

% node example.js
Server running at http://127.0.0.1:1337/

또한, 포트 번호 1337에서 수신하고 무엇이든 에코하는 간단한 TCP 서버는 다음과 같습니다. 

var net = require('net');

var server = net.createServer(function (socket) {
  socket.write("Echo server\r\n");
  socket.pipe(socket);
});

server.listen(1337, "127.0.0.1");

더 많은 예제가 보고 싶으신 분은 API 문서를 참고하세요.

NodeJS 관련 정보가 필요하신 분은 공식 위키(영문) 페이지를 참고하세요.

개요(@rhiokim님의 번역)

Node의 목표는 손쉽게 확장이 가능한 네트워크 프로그램을 만드는 방법을 제 공하는 것입니다. 위의 "Hello World" 웹 서버 예제에서는 다수의 클라이언트 와의 연결을 동시에 처리할 수 있습니다. Node는 OS에(epoll 이나 kqueue, /dev/poll 또는 select를 통해) 새로운 연결이 도착하면 알 수 있도록 지시 하고, 절전 상태가 됩니다. 그리고 새로운 연결이 있는 경우 콜백을 수행합니다. 각각의 연결은 아주 작은 힙 밖에 할당되지 않습니다.

이것은 OS 스레드를 가진 최근의 일반적인 동시성 모델과는 대조적입니다. 이 기술에 비해 쓰레드 기반 네트워크 프로그램은 비효율적이고 또한 아주 사용하기 어렵다는 것입니다. 여기 와 여기를 참조하십시오. Node는 하나의 연결마다 2MB의 스택 메모리를 할당하는 쓰레드 시스템에 비해 작업량이 매우 좋은 메모리 효율을 제공합니다. 또한 Node 사용자는 프로세스의 교착 상태에 대한 걱정에서 해방되었습니다. Node에는 잠금 장치가 없습니다. Node에 직접 I/O를 수행하는 함수가 거의 없기 때문에 프로세스는 결코 차단되지 않습니다. 블록이 없기 때문에 전문가가 아닌 프로그래머이라도 빠른 시스템을 개발하는 것이 가능합니다.

Node는 Ruby의 Event Machine이나 Python의 Twisted의 영향을 받고 있으며, 비슷한 디자인으로 되어있습니다. 그러나 Node의 이벤트 모델은 조금더 진보했습 니다. 즉 이벤트 루프를 라이브러리로 제공하는 것이 아니라 언어 자체에 포함된 구성으로 제공하고 있습니다. Node가 아닌 시스템에서는 이벤트 루프를 시작할 때 차단시키는 호출도 존재합니다. 그러한 시스템은 일반적으로 스크립트의 시작부분 에서 콜백을 통해 수행할 작업을 정의하고 스크립트의 마지막에 EventMachine::run() 과 같은 블록을 호출하여 서버를 시작하고 있습니다. Node 에서는 이러한 "이벤 트 루프의 시작" 호출하지 않습니다. Node는 입력되는 스크립트를 실행하고 나면 그대로 이벤트 루프에 들어갑니다. 그리고 동작해야할 콜백이 없으면 이벤트 루프 에서 빠져 나옵니다. 이러한 동작은 브라우저의 JavaScript와 동일합니다. 즉, 이벤트 루프는 사용자로부터 은닉되어 있는 것입니다.

HTTP는 Node에서 최고의 프로토콜 입니다. Node의 HTTP 라이브러리는 제작자의 웹 서버 개발 경험을 탈피하게 하였습니다. 예를 들어 데이터 스트리밍은 대부분의 웹 프레임워크에서는 불가능합니다. Node는 이러한 문제점을 HTTP 파서와 API를 통하여 해결하려 합니다. Node의 이벤트 중심적인 설치하는 경우 웹 라이브러리 및 프레임워크들을 위한 매우 좋은 기반이 될 것입니다.

하지만 멀티 프로세서 환경에서는 어떻게 될까? 멀티 코어 컴퓨터에서 프로그 램을 확장하기 위해서는 스레드를 사용해야 하는 것이 아닌가? 멀티코어 컴퓨터에서는 확장하기 위해 필요한 메모리 공유 쓰레드가 없습니다. 확장 가능한 시스템의 기본은 빠른 네트워킹 및 차단이 없는 설계이고 나머지는 메세지 전송이 있으면 충분합니다. 향후 버전에서 Node는 현재의 설계와 아주 잘 맞는 형태로 자식 프로세스를 fork ( Web Workers API 를 사용하여)할 수 있게 될 예정입니다.
